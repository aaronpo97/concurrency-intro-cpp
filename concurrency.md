_Standard: C++11 and later · Primary header group: `<thread>`, `<mutex>`,
`<condition_variable>`, `<future>`, `<atomic>`, `<semaphore>` (C++20), `<latch>`
/ `<barrier>` (C++20)_

## Foundational Concepts

**Thread** The smallest unit of execution within a process. Threads share the
same address space, permitting direct communication and synchronisation between
them ("std::thread," cplusplus.com). Managed via `std::thread` (`<thread>`).

**Concurrency vs. Parallelism** Concurrency is the property of a programme in
which several computations are executing during overlapping time periods, rather
than sequentially; it encompasses any situation in which distinct computations
can be in progress simultaneously. Parallelism implies literal simultaneous
execution across multiple processor cores; concurrency does not require it
("C++11 Library").

**Data Race** Occurs when two or more threads concurrently access the same
memory location, at least one access is a write, and no synchronisation
mechanism governs the access. Result: undefined behaviour.

**Critical Section** A code region that must not be executed by more than one
thread at a time.

**Deadlock** A state in which two or more threads are each waiting for a
resource held by another, resulting in indefinite blocking.

## Thread Management (`<thread>`)

| Component              | Description                                                                                                                                                                                                    |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `std::thread`          | Represents individual threads of execution — a sequence of instructions executable concurrently with others whilst sharing the same address space ("std::thread," cplusplus.com).                              |
| `.join()`              | Blocks the calling thread until the target thread finishes.                                                                                                                                                    |
| `.detach()`            | Separates the thread of execution from the `std::thread` object, allowing execution to continue independently; the thread is no longer joinable after this call ("std::thread," cplusplus.com).                |
| `std::jthread` (C++20) | A joining thread that, on destruction, first issues a stop request via `request_stop()` and then automatically calls `join()`. Also supports cooperative cancellation via stop tokens ("Concurrency Support"). |

## Mutual Exclusion (`<mutex>`)

**`std::mutex`** A lockable object designed to signal when critical sections of
code require exclusive access, preventing other threads with the same protection
from executing concurrently and accessing the same memory locations. Does not
support recursion — a thread shall not lock a mutex it already owns
("std::mutex").

**`std::recursive_mutex`** Variant of `std::mutex` permitting the same thread to
acquire the lock multiple times.

**`std::shared_mutex`** (C++17) Supports shared (read) and exclusive (write)
locking, enabling multiple concurrent readers.

**Lock Guards (RAII wrappers)**

| Type                       | Behaviour                                                                                                                                |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `std::lock_guard`          | Acquires mutex on construction, releases on destruction. Non-movable; no manual unlock.                                                  |
| `std::unique_lock`         | As `lock_guard`, but movable and supports deferred locking, timed locking, and manual `unlock()`. Required by `std::condition_variable`. |
| `std::shared_lock` (C++17) | Acquires shared (read) ownership of a `std::shared_mutex`.                                                                               |
| `std::scoped_lock` (C++17) | Acquires multiple mutexes simultaneously, deadlock-safe.                                                                                 |

## Condition Variables (`<condition_variable>`)

**`std::condition_variable`** A synchronisation primitive that allows multiple
threads to communicate with each other, permitting some number of threads to
wait — with an optional timeout — for notification from another thread that they
may proceed. Always associated with a mutex ("Concurrency Support").

`std::condition_variable` works exclusively with `std::unique_lock<std::mutex>`,
which allows for maximal efficiency on some platforms.
`std::condition_variable_any` provides equivalent functionality for any
BasicLockable object ("std::condition_variable," cppreference).

Protocol summary:

- **Notifying thread:** Acquires a `std::mutex` and modifies the shared variable
  while the lock is owned; then calls `notify_one` or `notify_all`. Notification
  may be issued after releasing the lock and is often preferable to do so, in
  order to avoid waking a waiting thread only for it to immediately re-block on
  the mutex ("std::condition_variable," cppreference).
- **Waiting thread:** Acquires a `std::unique_lock<std::mutex>` and calls
  `wait`, `wait_for`, or `wait_until`; the operation atomically releases the
  mutex and suspends the thread. Always guard `wait()` with a predicate to
  handle spurious wakeups ("std::condition_variable," cppreference).

## Atomic Operations (`<atomic>`)

**`std::atomic<T>`** Provides fine-grained atomic operations for lockless
concurrent programming. Each atomic operation is indivisible with regard to any
other atomic operation on the same object; atomic objects are free of data races
("Concurrency Support"). All atomic types except `std::atomic_flag` may be
implemented internally using mutexes or other locking mechanisms rather than
lock-free CPU instructions ("std::atomic<T>::is_lock_free," cppreference). Core
operations include `store`, `load`, and `exchange`.

**`std::atomic_flag`** An atomic boolean type and the only type guaranteed to be
lock-free across all implementations. Unlike `std::atomic<bool>`, it does not
provide `load` or `store` operations ("std::atomic_flag," cppreference). In
C++20, it additionally supports `wait()`, `notify_one()`, and `notify_all()`
("Concurrency Support").

**`std::memory_order`** An enumeration specifying the ordering constraints on
atomic operations (e.g., `memory_order_relaxed`, `memory_order_acquire`,
`memory_order_release`, `memory_order_seq_cst`). Controls visibility of writes
across threads.

## Futures, Promises & Async (`<future>`)

**`std::future<T>`** Provides a mechanism to access the result of asynchronous
operations. An asynchronous operation — performed via `std::async`,
`std::packaged_task`, or `std::promise` — can provide a `std::future` object to
the creator of that operation. Values or exceptions are communicated via a
shared state ("std::future," cppreference).

**`std::promise<T>`** The write ("push") end of the promise–future communication
channel. Stores a result or exception in the shared state, marks the state
ready, and unblocks any thread waiting on an associated future. Once a value or
exception has been set, it cannot be set again ("std::promise," cppreference).

**`std::shared_future<T>`** A copyable future permitting multiple threads to
observe the same shared state simultaneously. Access to the same shared state
from multiple threads is safe if each thread does so through its own copy of a
`std::shared_future` object ("std::shared_future," cppreference).

**`std::async`** Runs a function asynchronously — potentially in a separate
thread or thread pool — and returns a `std::future` that will hold the result of
that function call. The completion of the function is sequenced before making
the shared state ready ("std::async," cppreference).

**`std::packaged_task<T>`** Wraps any callable target so that it can be invoked
asynchronously. Its return value or any exception thrown is stored in a shared
state accessible through an associated `std::future` object
("std::packaged_task," cppreference).

> **Note:** A `std::promise` has a single shared state: `get_future()` may be
> called at most once, and a value or exception may be set at most once.
> Similarly, `std::future::get()` may only be called once; subsequent calls
> throw `std::future_error`. For repeated inter-thread communication, use
> condition variables or atomics ("std::promise," cppreference; "std::future,"
> cppreference).

## Semaphores, Latches & Barriers (C++20)

**`std::counting_semaphore` / `std::binary_semaphore`** (`<semaphore>`) A
lightweight synchronisation primitive used to constrain concurrent access to a
shared resource. When either would suffice, a semaphore can be more efficient
than a condition variable ("Concurrency Support").

**`std::latch`** (`<latch>`) A thread coordination mechanism that allows any
number of threads to block until an expected number of threads arrive. A latch
cannot be reused ("Concurrency Support").

**`std::barrier`** (`<barrier>`) A reusable thread coordination mechanism.
Unlike `std::latch`, once a group of arriving threads are unblocked the barrier
may be reused for a subsequent phase. Barriers also execute a (possibly empty)
completion callable before unblocking threads ("std::barrier," cppreference).

---

## Stop Tokens (C++20, `<stop_token>`)

**`std::stop_source` / `std::stop_token` / `std::stop_callback`** Components
used to asynchronously request that an operation stops execution in a timely
manner, typically because the result is no longer required. `std::stop_source`
controls and publishes the stop state; `std::stop_token` provides a thread-safe,
read-only view of that state; `std::stop_callback` registers a callback to be
invoked upon a stop request ("Concurrency Support").

## Common Pathologies

| Term                | Definition                                                                                                      |
| ------------------- | --------------------------------------------------------------------------------------------------------------- |
| **Race condition**  | Incorrect programme behaviour arising from unsynchronised concurrent access to shared state.                    |
| **Deadlock**        | Circular waiting on mutexes held by other blocked threads.                                                      |
| **Livelock**        | Threads actively respond to each other without making progress.                                                 |
| **Starvation**      | A thread is perpetually denied access to a resource.                                                            |
| **Spurious wakeup** | A condition variable wait returns without a corresponding notification; always guard `wait()` with a predicate. |
